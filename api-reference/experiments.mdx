---
title: "Experiments API endpoints"
description: "Create, list, archive, and retrieve hypothesis experiments through the Oriyn REST API. Get verdicts, convergence scores, and per-persona response breakdowns."
---

Experiments let you test a hypothesis against your enriched personas. You submit a hypothesis in plain language, and Oriyn evaluates it against persona-grounded simulated agents — returning a verdict, a convergence score, a per-persona breakdown, and per-question response distributions.

<Note>
Enrichment must be complete before you can create experiments. Ensure `enrichment_status` is `ready` on your product first, and that personas have populated `source_users` (re-run enrichment if not).
</Note>

## Endpoints

- [List experiments](#list-experiments) — `GET /v1/products/{id}/experiments`
- [Create an experiment](#create-an-experiment) — `POST /v1/products/{id}/experiments`
- [Get an experiment](#get-an-experiment) — `GET /v1/products/{id}/experiments/{experiment_id}`
- [Archive an experiment](#archive-an-experiment) — `PATCH /v1/products/{id}/experiments/{experiment_id}/archive`

---

Retrieve all experiments for a product.

```
GET /v1/products/{id}/experiments
```

<ParamField path="id" type="string" required>

### Response fields

<ResponseField name="id" type="string" required>
  Unique identifier for the experiment.
</ResponseField>

<ResponseField name="title" type="string">
  Auto-generated short title (3–6 words) summarizing the hypothesis. `null` if title generation has not completed yet.
</ResponseField>

<ResponseField name="hypothesis" type="string" required>
  The hypothesis text that was submitted.
</ResponseField>

<ResponseField name="status" type="string" required>
  Current status: `processing`, `complete`, or `failed`.
</ResponseField>

<ResponseField name="verdict" type="string">
  The outcome: `ship`, `revise`, or `reject`. `null` while the experiment is still processing.
</ResponseField>

<ResponseField name="convergence" type="number">
  Convergence score between `0` and `1` indicating how strongly agents agreed across the survey questions. `null` while processing.
</ResponseField>

<ResponseField name="created_by_email" type="string" required>
  Email of the user who created the experiment.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  ISO 8601 timestamp of when the experiment was created.
</ResponseField>

## Create an experiment

Submit a new hypothesis for evaluation. Oriyn evaluates the hypothesis against your product's personas — sampling source users from each persona, building per-agent prompts, and running them in parallel — then aggregates the results into a verdict.

```
POST /v1/products/{id}/experiments
```

This is an async operation. The endpoint returns `202 Accepted` immediately. Poll `GET /v1/products/{id}/experiments/{experiment_id}` and check `status` until it is `complete` or `failed`.

<Note>
Requires the `org:content:write` permission on the organization that owns the product. Owners, admins, and members have this permission by default; viewers do not. See [authorization errors](/api-reference/overview#authorization-errors) for the `403` response shape.
</Note>

### Path parameters

<ParamField path="id" type="string" required>
  The product UUID.
</ParamField>

### Request body

<ParamField body="hypothesis" type="string" required>
  The hypothesis to evaluate. Write it as a plain-language statement of what you believe to be true and why. The more specific, the better.
</ParamField>

<ParamField body="agent_count" type="integer">
  Number of simulated agents to run (1–1000). Defaults to the server-configured value. Higher counts produce more statistically robust results but cost more credits.

  Each plan has a maximum agent count. Enterprise organizations may have a custom ceiling via an org-level override. Billing-exempt organizations have no limit.
</ParamField>

### Error responses

Creating an experiment can return additional error codes beyond the standard set:

| Status | Error | Description |
|---|---|---|
| `402 Payment Required` | `insufficient_credits` | Your organization doesn't have enough credits. The response includes `credits_required`, `credits_available`, and `plan`. |
| `403 Forbidden` | `agent_count_exceeded` | The requested `agent_count` exceeds your plan's effective maximum. The response includes `max_agent_count` and `plan`. |
| `403 Forbidden` | Authorization failure | The caller's role lacks the `org:content:write` permission on the org that owns this product. See [authorization errors](/api-reference/overview#authorization-errors). |

```json 402 — insufficient credits
{
  "error": "insufficient_credits",
  "credits_required": 10,
  "credits_available": 1,
  "plan": "free"
}
```

```json 403 — agent count exceeded
{
  "error": "agent_count_exceeded",
  "max_agent_count": 100,
  "plan": "free"
}
```

<Note>
Billing-exempt organizations bypass both credit checks and agent count limits. Enterprise organizations may have custom agent count ceilings that differ from their plan's default.
</Note>

<CodeGroup>

```bash cURL
curl --request POST \
  https://api.oriyn.ai/v1/products/<product-id>/experiments \
  --header "Authorization: Bearer <your-token>" \
  --header "Content-Type: application/json" \
  --data '{
    "hypothesis": "Adding a guided setup wizard will reduce onboarding drop-off by making the data connection step feel less daunting."
  }'
```

</CodeGroup>

```json Response — 202 Accepted
{
  "experiment_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

<Tip>
Poll every 2–5 seconds. Most experiments complete within 30 seconds.
</Tip>

## Get an experiment

Retrieve a single experiment by ID, including the full result when complete.

```
GET /v1/products/{id}/experiments/{experiment_id}
```

### Path parameters

<ParamField path="id" type="string" required>
  The product UUID.
</ParamField>

<ParamField path="experiment_id" type="string" required>
  The experiment UUID.
</ParamField>

<CodeGroup>

```bash cURL
curl https://api.oriyn.ai/v1/products/<product-id>/experiments/<experiment-id> \
  --header "Authorization: Bearer <your-token>"
```

</CodeGroup>

  "created_by_email": "maya@example.com",

### Response fields

<ResponseField name="id" type="string" required>
  Unique identifier for the experiment.
</ResponseField>

<ResponseField name="product_id" type="string" required>
  ID of the product this experiment belongs to.
</ResponseField>

<ResponseField name="hypothesis" type="string" required>
  The hypothesis that was evaluated.
</ResponseField>

<ResponseField name="status" type="string" required>
  Current status: `processing`, `complete`, or `failed`.
</ResponseField>

<ResponseField name="created_by_email" type="string" required>
  Email of the user who created the experiment.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  ISO 8601 timestamp of when the experiment was created.
</ResponseField>

<ResponseField name="summary" type="object">
  The experiment result. `null` while `status` is `processing`.
  <Expandable title="summary fields">
    <ResponseField name="verdict" type="string" required>
      Overall recommendation: `ship`, `revise`, or `reject`. Derived from the agent-weighted distribution of `adoption_likelihood`.
    </ResponseField>
    <ResponseField name="convergence" type="number" required>
      Convergence score between `0` and `1`. Computed as the average normalized inverse entropy across all survey questions — higher values indicate stronger agreement among agents.
    </ResponseField>
    <ResponseField name="summary" type="string" required>
      Plain-language explanation of the verdict, including the share of agents that would and would not adopt the change.
    </ResponseField>
    <ResponseField name="persona_breakdown" type="object[]" required>
      Per-persona evaluation results.
      <Expandable title="persona breakdown fields">
        <ResponseField name="persona" type="string" required>
          Name of the persona.
        </ResponseField>
        <ResponseField name="response" type="string" required>
          Dominant sentiment for this persona: `positive`, `neutral`, or `negative`. Derived from agents' `primary_reaction` answers (e.g., `excited`/`interested` map to `positive`).
        </ResponseField>
        <ResponseField name="reasoning" type="string" required>
          Representative reasoning quote selected from this persona's agents (the response closest to the persona's mean term-frequency vector).
        </ResponseField>
        <ResponseField name="adoption_rate" type="number" required>
          Weighted share of this persona's agents that answered `definitely` or `probably` to the adoption question. Value between `0` and `1`, rounded to three decimals.
        </ResponseField>
      </Expandable>
    </ResponseField>
    <ResponseField name="question_results" type="object" required>
      Per-question response distributions. Keys are question IDs (such as `adoption_likelihood`, `primary_reaction`, `behavior_change`, `concern_level`); each value contains a `distribution` map of option → weighted count and a `total_weight`.
    </ResponseField>
    <ResponseField name="agent_count" type="integer" required>
      Number of agents whose responses were aggregated into this summary.
    </ResponseField>
  </Expandable>
</ResponseField>

## Archive an experiment

Mark an experiment as archived. Archived experiments are hidden from the default list view in the web app but remain retrievable by ID.

```
PATCH /v1/products/{id}/experiments/{experiment_id}/archive
```

### Path parameters

<ParamField path="id" type="string" required>
  The product UUID.
</ParamField>

<ParamField path="experiment_id" type="string" required>
  The experiment UUID.
</ParamField>

<CodeGroup>

```bash cURL
curl --request PATCH \
  https://api.oriyn.ai/v1/products/<product-id>/experiments/<experiment-id>/archive \
  --header "Authorization: Bearer <your-token>"
```

</CodeGroup>

```json Response
{
  "status": "archived"
}
```
