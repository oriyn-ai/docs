The Oriyn API is a REST API that gives you programmatic access to your behavioral intelligence data. All requests go to:

```
https://api.oriyn.ai
```

All authenticated endpoints live under the `/v1` prefix. All requests and responses use JSON. Include `Content-Type: application/json` on any request that sends a body.

## Authentication

Every endpoint (except the health check at `/health` and the discovery root at `/v1/`) requires a Bearer token in the `Authorization` header:

```bash
Authorization: Bearer <your-token>
```

See [Authentication](/api-reference/authentication) for details on obtaining and managing tokens.

**Verify your token** by calling `GET /v1/me`:

<CodeGroup>

```bash cURL
curl https://api.oriyn.ai/v1/me \
  --header "Authorization: Bearer <your-token>"
```

</CodeGroup>

```json Response
{
  "user_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "email": "you@example.com"
}
```

## Async operations

Several operations take time to complete. These endpoints return HTTP `202 Accepted` immediately and run in the background:

| Endpoint | What it does |
|---|---|
| `POST /v1/products/{id}/context` | Synthesizes product context from your URLs, scraped sources, and uploaded documents |
| `POST /v1/products/{id}/enrich` | Runs behavioral enrichment to generate personas |
| `POST /v1/products/{id}/hypotheses/refresh` | Re-mines behavioral hypotheses and bottlenecks in the background |
| `POST /v1/products/{id}/experiments` | Evaluates a hypothesis against your personas |

After receiving a `202`, poll the relevant status field to track progress:

- **Context:** check `context_status` on `GET /v1/products/{id}` — values: `idle`, `processing`, `ready`, `failed`
- **Enrichment:** check `enrichment_status` on `GET /v1/products/{id}` — values: `idle`, `processing`, `ready`, `failed`
- **Experiments:** check `status` on `GET /v1/products/{id}/experiments/{experiment_id}` — values: `processing`, `complete`, `failed`

<Tip>
Poll at a reasonable interval — every 2–5 seconds is sufficient for most operations. Avoid tight loops with no delay.
</Tip>

## Error responses

All error responses use standard HTTP status codes and return a JSON body describing the problem.

| Status | Meaning |
|---|---|
| `400 Bad Request` | Invalid input (e.g., source URL must use `https://`) |
| `401 Unauthorized` | Missing or invalid Bearer token, or the authenticated user has no active organization |
| `402 Payment Required` | Insufficient credits to complete the operation |
| `403 Forbidden` | Your token is valid but you don't have access to this resource, or you've exceeded a plan limit |
| `404 Not Found` | The resource doesn't exist or doesn't belong to your organization |
| `409 Conflict` | The operation is already in progress (e.g., enrichment is already running) |
| `422 Unprocessable Entity` | The request is well-formed but cannot be processed in the current state (e.g., enrichment requested before context is ready) |

## Authorization errors

Some endpoints require a specific permission on the organization that owns the resource. The caller's role on their currently active organization is resolved when the token is validated, and permissions are derived from that role. When the caller is authenticated but their role does not grant the required permission, the API returns `403 Forbidden` with a structured body:

```json 403 — missing permission
{
  "error": "this action requires `org:content:write`",
  "required_permission": "org:content:write",
  "role": "org:viewer"
}
```

If the caller has no organization membership (for example, the user has not yet completed onboarding), the API returns `401 Unauthorized` instead. The frontend should redirect to onboarding to complete org setup.

### Roles and permissions

Each organization member has one of four roles. The role determines which permissions they hold. Both roles and permissions are scoped under the `org:` namespace:

| Permission | `org:owner` | `org:admin` | `org:member` | `org:viewer` |
|---|:-:|:-:|:-:|:-:|
| `org:content:write` — create and modify products, experiments | Yes | Yes | Yes | — |
| `org:sys_billing:manage` — manage subscription, cards, and credit purchases | Yes | — | — | — |

Endpoint reference pages call out the specific permission each mutating endpoint requires. Members and roles are managed in the [Oriyn web app](https://app.oriyn.ai); the API is not the source of truth for membership.

## Discovering the API

The API exposes a self-describing root at `GET /v1/` that returns a curated, LLM-friendly summary of every resource and a link to the full OpenAPI spec at `/openapi.json`. This endpoint is unauthenticated by design — discovery should not require a key.
