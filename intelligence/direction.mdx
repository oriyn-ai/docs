<Warning>
The `direction` API endpoint has been removed. The `GET /products/{product_id}/direction` endpoint and the `oriyn direction` CLI command are deprecated and no longer return data. Equivalent insight is now produced by combining [hypotheses](/intelligence/patterns), [bottlenecks](/api-reference/enrichment#get-bottlenecks), and [experiments](/experiments/overview) — let the data tell you what to fix, then validate the fix against your personas before you ship it.
</Warning>

## How to get equivalent insight today

The previous direction output bundled three things into one record: behavioral signals, prioritized actions, and rationale. Each piece is now reachable as a first-class API resource.

### 1. Find what's worth fixing

Start with the data Oriyn has already mined for you:

- **Hypotheses** — recurring behavioral sequences with frequency and significance scores. See [Get hypotheses](/api-reference/enrichment#get-hypotheses).
- **Bottlenecks** — the slowest directly-follows edges in your behavioral graph, ranked by mean transit time and the number of users affected. See [Get bottlenecks](/api-reference/enrichment#get-bottlenecks).

Both endpoints return `source_users` — sample identifiers of the people who actually exhibited the behavior. That's your audit trail.

### 2. Validate before you build

Once you have a candidate change in mind, run an experiment to check it against your personas before committing to the work:

- See [Create an experiment](/api-reference/experiments#create-an-experiment) for the API.
- The verdict (`ship`, `revise`, or `reject`), convergence score, and per-persona breakdown stand in for the previous "priority" field — grounded in agent-weighted simulation rather than a single LLM-assigned label.

### 3. Wire it together

A typical flow that replaces the old direction call:

1. `GET /v1/products/{id}/hypotheses` — surface high-significance sequences.
2. `GET /v1/products/{id}/bottlenecks` — find the slow edges that frustrate users most.
3. Pick the highest-leverage candidate, write a hypothesis, and `POST /v1/products/{id}/experiments` to validate it.
4. Read the experiment's `summary` for verdict, convergence, and per-persona reasoning.

This produces an evidence-backed action plan with stronger grounding than the previous direction endpoint provided.
