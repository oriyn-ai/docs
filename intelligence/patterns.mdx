---
title: "Behavioral Patterns"
description: "Recurring behavioral sequences found in your user data — the workflows users actually follow, not the ones you designed."
---

Oriyn scans your PostHog event data and extracts 3–7 recurring behavioral sequences per enrichment run. These are patterns that appear consistently across users: multi-step flows that users repeat, paths that lead to core actions, and sequences that end in drop-off. Unlike personas, which describe segments of users, patterns describe what users *do*.

## What a pattern contains

Each pattern includes:

- **`title`** — a short name for the pattern (e.g., "Import-then-abandon", "Explore before commit")
- **`description`** — what users do in this sequence and why it matters for your product
- **`raw_sequence`** — an ordered list of steps that define the pattern, derived from real event data
- **`frequency`** — how often this pattern appears across your user base
- **`significance`** — a plain-language explanation of what this pattern means for product decisions
- **`generated_at`** — the timestamp for when this pattern was generated

The `raw_sequence` is the most precise part of the pattern. It shows the actual ordered steps users take, in the order they take them, so you can trace the behavior back to specific parts of your product.

## Retrieve patterns

### API

```bash
curl https://api.oriyn.ai/products/{product_id}/patterns \
  -H "Authorization: Bearer <token>"
```

### CLI

```bash
oriyn patterns --product-id <product_id>
```

Add `--json` to get the raw JSON response:

```bash
oriyn patterns --product-id <product_id> --json
```

## Typical use cases

**Discovering unmapped workflows.** Users often find paths through your product that you didn't explicitly design. Patterns surface these so you can understand, support, or redirect them.

**Spotting drop-off sequences.** If a pattern consistently ends before a conversion event, that's a structured description of where users are falling off — and what led them there.

**Prioritizing friction to fix.** When a pattern is both frequent and ends in drop-off, it represents concentrated, addressable loss. The `significance` field gives you a starting interpretation, but the `raw_sequence` lets you verify it against your own product knowledge.

Patterns update each time enrichment runs. Re-running enrichment after a product change will reflect whether the behavioral sequence has shifted.
