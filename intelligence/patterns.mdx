Oriyn mines recurring behavioral sequences from your cross-provider event data and surfaces them as **hypotheses** — testable sequences that appear consistently across users. These are multi-step flows that users repeat, paths that lead to core actions, and sequences that end in drop-off. Unlike personas, which describe segments of users, hypotheses describe what users *do*.

<Note>
The API field this resource was previously called "patterns" is now exposed as `hypotheses`. Existing documentation may reference the older name; the canonical endpoint is `GET /v1/products/{id}/hypotheses`. See the [Enrichment API reference](/api-reference/enrichment#get-hypotheses) for the full schema.
</Note>

## What a hypothesis contains

Each hypothesis includes:

- **`sequence`** — the ordered list of raw activity names that define the behavior, derived from real event data
- **`rendered_sequence`** — the same activities rendered as human-readable labels
- **`frequency`** — total number of times this sequence was observed across all users in the mined window
- **`user_count`** — number of distinct users who exhibited the sequence
- **`significance_pct`** — significance score, expressed as a percentage of the user population
- **`source_users`** — sample of resolved user identifiers that exhibited the sequence — useful for seeding experiments or drilling into a cohort

The `sequence` is the most precise part of the hypothesis. It shows the actual ordered steps users take, in the order they take them, so you can trace the behavior back to specific parts of your product.

## Retrieve hypotheses

### API

```bash
curl https://api.oriyn.ai/v1/products/{product_id}/hypotheses \
  -H "Authorization: Bearer <token>"
```

**Response:**

```json
{
  "enrichment_status": "ready",
  "data": [
    {
      "sequence": [
        "started_integration_setup",
        "reached_credentials_step",
        "navigated_away",
        "returned_to_integration_setup",
        "reached_credentials_step"
      ],
      "rendered_sequence": [
        "Started integration setup",
        "Reached credentials step",
        "Navigated away",
        "Returned to integration setup",
        "Reached credentials step"
      ],
      "frequency": 412,
      "user_count": 287,
      "significance_pct": 18.4,
      "source_users": [
        "user_01H8ZK3P...",
        "user_01H92YTM..."
      ]
    }
  ]
}
```

To trigger a fresh mining pass before reading, call `POST /v1/products/{product_id}/hypotheses/refresh`. See [Refresh hypotheses](/api-reference/enrichment#refresh-hypotheses).

### CLI

```bash
oriyn hypotheses --product-id <product_id>
```

Add `--json` to get the raw JSON response:

```bash
oriyn hypotheses --product-id <product_id> --json
```

## Typical use cases

**Discovering unmapped workflows.** Users often find paths through your product that you didn't explicitly design. Hypotheses surface these so you can understand, support, or redirect them.

**Spotting drop-off sequences.** If a sequence consistently ends before a conversion event, that's a structured description of where users are falling off — and what led them there.

**Prioritizing friction to fix.** When a sequence is both frequent and ends in drop-off, it represents concentrated, addressable loss. Use `source_users` to seed an experiment that tests a fix against the exact cohort that exhibited the behavior.

Hypotheses update each time the pattern-miner runs. Re-running mining after a product change will reflect whether the behavioral sequence has shifted.
